# 项目计划

## 1. 项目目标
在课程截止时间前，完成一个可复现、可讲解、可做消融分析的子图匹配项目，满足选题 2 的全部必做要求，并为选做内容预留扩展空间。

## 2. 交付物清单
- 子图匹配文献调研与分类体系
- 1 篇指定论文的复现实现
- 3-4 组数据集实验结果与分析
- 过滤策略消融实验结果
- 英文实验报告
- 汇报 PPT 与答辩准备材料
- 可复现实验脚本、绘图脚本和运行说明

## 3. 阶段计划

### 阶段 A：选题收敛与论文确定
目标：完成论文选择，并将后续实现路线固定到目标论文。

任务：
- 阅读候选论文 `[13] [14] [15] [16] [17]` 的摘要、方法和实验部分
- 比较复现难度、数据集可得性、过滤策略是否适合做消融
- 选定目标论文，并输出 1 份简短比较结论

完成标志：
- 已选定目标论文 `GUP`
- 已有论文选择说明文档
- 已形成 `GUP` 的复现规划和模块拆分草案

### 阶段 B：数据与接口规范
目标：统一后续代码和实验的输入输出格式。

任务：
- 定义数据图与查询图文件格式
- 约定标签存储方式、边格式、图是否无向
- 定义实验输出字段
- 定义 `result mappings`、`partial mappings`、`pruned partial mappings` 的计数口径

完成标志：
- 图加载接口固定
- 指标定义写入文档
- 样例数据可用于调试

### 阶段 C：基础匹配框架实现
目标：先做出正确、可测、可统计的基础版本。

任务：
- 实现图数据结构与解析器
- 实现候选集生成
- 实现基础匹配顺序
- 实现回溯搜索
- 接入基础计数器与日志

完成标志：
- 可在 toy graph 上返回正确匹配结果
- 能输出基础统计信息

### 阶段 D：论文方法复现
目标：把 `GUP` 中的关键策略真正落到代码上。

任务：
- 拆解 `GUP` 中的 `reservation guard`、`nogood guard`、guarded candidate space 和搜索节点编码
- 将 guard 设计为独立模块或开关
- 记录不同 guard 剪掉的 `partial mappings` 数量
- 保留一套关闭 guard 后的对照流程
- 明确哪些内容属于必做复现，哪些优化属于可选增强

完成标志：
- `GUP` 核心方法跑通
- 各类 guard 可单独启停
- 消融实验可以通过配置驱动
- 统计结果能对应课程要求的四类指标

补充说明：
- 已调查代码可得性，当前未发现 `GUP` 的明确公开仓库，因此本阶段默认采用“自行实现课程版”的路线。
- `VF3` 有官方源码入口，但需要注册并申请下载；若 `GUP` 实现风险超出预期，`VF3` 可作为可执行备选。

### 阶段 E：实验与报告
目标：完成最终交付物。

任务：
- 选择 3-4 组数据集与查询集
- 批量运行实验并保存结果
- 绘制图表并生成报告表格
- 撰写英文报告
- 准备汇报 PPT 和答辩问答

完成标志：
- 必做实验全部完成
- 图表与表格可直接用于报告
- 仓库具备复现说明

## 6. 当前论文决策
当前已正式确定复现论文为 `[14] GUP`，主要原因如下：

1. guard 机制边界清晰，适合做模块化实现
2. 与课程要求中的过滤策略消融天然契合
3. 便于统计 `partial mappings` 和 `pruned partial mappings`
4. 适合二人小组在有限时间内完成
5. 适合 5-7 分钟课堂汇报，容易讲清楚方法贡献

补充决策说明：
- 从“课程任务匹配度”看，`GUP` 仍是首选。
- 从“现成代码可获得性”看，`VF3` 更有优势，因为官方提供源码申请入口。
- 当前没有查到 `GUP` 的明确公开实现，因此项目默认按“自己实现 `GUP-lite`”推进。

## 7. 风险与应对
- 论文过难：优先选策略边界清晰的论文，避免复杂索引或过重工程依赖
- 数据难找：优先选择公开数据集和已有格式说明的论文
- 指标口径不统一：先写文档，再写代码
- 实验耗时过长：先用小图验证正确性，再扩大到正式数据集
- 后期报告赶工：实验脚本从一开始就保存可复用表格和图片
- `GUP` guard 逻辑理解偏差：先实现简化版，再逐步向论文细节靠拢
- `nogood` 相关状态记录过重：先保证正确性和可解释性，再看是否需要紧凑编码优化
- `GUP` 现成代码不可得：预留 `VF3` 作为切换备选，避免后期无路可退
- `VF3` 源码需注册申请：若要切换到 `VF3`，应尽早完成账号注册和代码申请
