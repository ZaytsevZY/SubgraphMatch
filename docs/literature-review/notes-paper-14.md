# 论文 `[14]` 阅读笔记

## 1. 基本信息
- 题目：`GuP: Fast Subgraph Matching by Guard-based Pruning`
- 作者：`Junya Arai, Yasuhiro Fujiwara, Makoto Onizuka`
- 出处：`Proc. ACM Manag. Data 2023`
- 当前定位：`课程项目最终选定复现论文`

## 2. 论文解决什么问题
论文关注的是精确子图匹配中的无效回溯问题。

即使已有方法会先做候选顶点和候选边过滤，搜索过程中仍然会出现大量“看起来可扩展，但最终一定失败”的部分映射。这些无效递归会显著拖慢运行时间。

GUP 的核心目标是：

- 不只是提前构造一个静态候选空间
- 而是在搜索过程中，根据当前部分映射状态，动态过滤掉不可能通向完整匹配的扩展

## 3. 核心算法是什么
GUP 的核心是 `guard-based pruning`。

可以把 `guard` 理解成“某类搜索状态一旦出现，就说明当前候选扩展不值得继续尝试”的规则。论文把 guard 附加在候选顶点和候选边上，在回溯过程中动态生效。

论文中的关键组成包括：

### 3.1 Guarded Candidate Space
- 在普通候选空间基础上，给候选顶点和候选边附加 guard 信息
- 后续搜索时不仅检查标签、边一致性和单射约束，还要检查 guard 是否触发

### 3.2 Reservation Guard
- 用来提前传播单射约束
- 直观理解：如果某个候选顶点在后续扩展中“预留”了一些数据顶点，而当前部分映射已经占用了这些顶点，那么这个候选扩展就应该被提前剪掉

### 3.3 Nogood Guard
- 从已经遇到的失败搜索状态中学习冲突模式
- 当当前部分映射包含某个已知的冲突模式时，可以直接剪枝，而不必再次把整棵失败子树走一遍

### 3.4 Search-node Encoding
- 用更紧凑的方式表示 guard 依赖的搜索状态
- 这部分更偏优化，可放在第一版正确性实现之后再做

## 4. 有哪些过滤或加速策略
按课程项目可实现性，GUP 可以分成以下几层：

### 4.1 必做层
- 基础候选集过滤
- 基础匹配顺序
- Guarded candidate space 的数据结构表达
- Reservation guard 剪枝
- Nogood guard 剪枝

### 4.2 建议做层
- Guard 命中次数统计
- 不同 guard 类型的剪枝计数
- 用配置开关控制不同 guard 是否启用

### 4.3 可选增强层
- 搜索节点紧凑编码
- 并行化
- 与论文实现更接近的候选空间构造细节

## 5. 为什么适合课程项目复现
GUP 非常适合本课程项目，主要原因如下：

- guard 是清晰可讲的核心创新点
- 可以自然拆成多个可开关模块，便于做消融
- 与课程要求的 `pruned partial mappings` 统计高度一致
- 与当前仓库已有的 baseline DFS/backtracking 框架很容易结合

相比之下，它比复杂索引型方法更适合 `Python` 原型推进，也比纯经典算法更容易体现“论文贡献”。

## 6. 数据集是否容易获得
从公开信息看，GUP 使用的是标准子图匹配实验范式，应该包含公开图数据和查询集。

当前仍需结合论文原文进一步确认：

- 具体使用了哪些数据集
- 查询图规模如何划分
- 是否有公开查询生成方式
- 是否设置了统一超时阈值

当前判断：

- 数据集总体可得性：`中高`
- 查询集获取难度：`中`
- 后续动作：优先从论文实验部分整理出可复用的 `3-4` 组数据集组合

## 7. 能否自然支持消融分析
非常适合。

建议最少做以下消融：

1. `Baseline`：无 guard，只保留基础候选过滤和回溯
2. `+ Reservation Guard`
3. `+ Nogood Guard`
4. `+ Reservation + Nogood`

如果后续实现了更细粒度版本，还可以继续拆：

1. 只关顶点 guard
2. 只关边 guard
3. 关闭某类 nogood 更新规则
4. 关闭搜索节点编码优化

## 8. 对当前仓库的映射建议
建议把 GUP 映射到当前工程结构如下：

- `matching/candidate.py`：基础候选集
- `matching/order.py`：基础顶点顺序
- `filters/`：guard 相关逻辑
- `matchers/backtracking_matcher.py` 或新建 `gup_matcher.py`：搜索主体
- `metrics/stats.py`：记录不同 guard 的 prune 次数

建议新增的统计字段包括：

- `pruned_by_reservation_guard`
- `pruned_by_nogood_guard_vertex`
- `pruned_by_nogood_guard_edge`
- `guard_checks_total`

## 9. 当前复现策略
当前不建议一步到位追求论文中的全部优化细节，而建议采用分层推进：

### 第一阶段：GUP-lite
- 复用当前 baseline 回溯框架
- 加入基础 guard 接口
- 先实现可解释的 reservation / nogood 剪枝
- 先保证计数和正确性

当前阶段补充：
- `reservation_guard` 当前实现更接近“可解释的前向检查近似”。
- `nogood_guard` 当前实现已经从“精确前缀 memo”改为“未来子问题 memo”：它记录某次扩展之后的 `used data vertices + frontier mapping + depth/extension identity`，从而可以在不同但等价的前缀之间复用 dead-end 信息。
- 这仍然是课程版 `GUP-lite`，不是论文工程实现的逐字还原，但已经更接近 guard 的核心思想。

### 第二阶段：课程实验版
- 增加开关式消融配置
- 输出结构化实验结果
- 准备 `3-4` 组数据集实验

### 第三阶段：增强版
- 尝试加入更贴近论文的状态编码或其他优化
- 如时间允许，再考虑并行化或更强候选空间构造

## 10. 当前结论
- 论文 `[14] GUP` 是当前最适合本项目的复现对象
- 第一版目标不应是“完整还原所有工程细节”，而应是“先做出正确、可统计、可消融的 GUP 核心版本”
- 后续实现应优先保证：
- 正确性
- guard 剪枝统计
- 消融开关
- 批量实验可复现
