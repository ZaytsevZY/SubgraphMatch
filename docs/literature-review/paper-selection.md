## 2. 评分口径
为方便统一比较，本文档使用 `1-5` 分制，`5` 分表示越适合课程项目：

- 方法清晰度：论文主线是否容易讲清楚
- 实现可控性：是否适合在课程周期内完成一个可运行版本
- 数据/实验可得性：数据集、查询集、公开资料是否容易找到
- 消融友好度：是否能自然拆成若干可开关模块
- 指标对齐度：是否容易统计 `runtime / result mappings / partial mappings / pruned partial mappings`
- 两人小组适配度：是否适合当前小组规模和时间

## 4. 逐篇整理

### 4.1 `[13] Efficient Subgraph Matching`

**简介**

- 这是一个较完整的现代精确子图匹配框架，核心目标不是只优化某一个过滤器，而是把 `dynamic programming`、`adaptive matching order` 和 `failing set` 协同起来。
- 从方法定位上看，它试图同时解决候选空间缩减、搜索顺序优化和失败后快速回跳的问题，因此更像一篇“系统化组合优化”的论文，而不是单点技巧论文。

**实验规模**

- 按论文定位判断，实验应属于“多数据集 + 多查询集 + 与主流精确匹配算法对比”的完整评测风格。
- 当前已确认其为 `SIGMOD 2019` 论文，但本文档尚未补到原文中的完整实验表，因此具体使用了哪些数据集、查询规模、超时设置，后续仍需结合原文实验部分确认。

**实验情况**

- 预期实验重点应包括：整体运行时间、难查询上的性能表现、不同优化组件组合后的增益。
- 由于它是组合式框架，理论上也较适合做“关闭某一组件”的消融，但真正实现时需要先把几个部分都搭起来。

**适合复现程度**

- 优点：技术路线完整，做出来后报告内容会比较扎实。
- 风险：并不是一个“只加一个过滤器”的方法，实现面较宽，容易在候选空间构造、匹配顺序和失败集维护三处同时遇到工程难点。

### 4.2 `[14] GUP`

**简介**

- GUP 的核心是 `guard-based pruning`。它给候选顶点和候选边附加 guard，根据当前部分映射的状态，自适应过滤掉不会导向正确结果的扩展。
- 其核心 guard 主要包括 `reservation guard` 和 `nogood guard`，方法主线清晰，非常适合课程项目做模块化复现。

**实验规模**

- 从公开摘要和论文结构可知，实验部分至少包含：
- 与已有方法的性能比较
- 平均处理时间
- 递归次数或搜索规模对比
- 各类 guard 的效果分析
- 内存占用分析
- 并行化效果分析

**实验情况**

- 论文的结论方向非常明确：guard 能显著减少无效递归和搜索空间，并能处理一些现有 SOTA 在实际时间限制内无法完成的困难查询。
- 对课程项目来说，这种“方法贡献和实验现象一一对应”的论文最容易讲清楚。

**适合复现程度**

- 优点：边界清楚，容易拆模块，适合做开关式消融。
- 风险：仍然需要理解 guard 的构造逻辑和状态编码，但整体风险明显低于索引型和分解型论文。

### 4.3 `[15] BEE`

**简介**

- BEE 的核心目标是减少回溯中的重复搜索。它通过 `block-separator decomposition` 在更大范围内识别冗余搜索，而不是只做局部或同层冗余去重。
- 与传统方法相比，BEE 更强调“全局范围 + 部分层级”的冗余检测能力。

**实验规模**

- 已确认论文发表于 `Proc. ACM Manag. Data 2025`。
- 公开摘要表明作者做了较充分的实验，并强调在 `EPS (embeddings per second)` 指标下取得了数量级上的加速。
- 但目前本文档尚未整理出它的完整数据集清单与查询配置，后续仍需从原文实验部分补全。

**实验情况**

- 论文摘要给出的结论很强：相对已有 SOTA，在 `EPS` 指标下有多数量级提速。
- 但该论文更偏“redundancy reduction”视角，不是直接围绕课程要求中的 `pruned partial mappings` 展开，因此课程指标需要自行映射。

**适合复现程度**

- 优点：新、强、思路有亮点。
- 风险：论文很新，现成资料和复现经验通常更少；方法本身涉及 block-separator tree 和 block reference 结构，对二人小组来说有一定实现压力。

### 4.4 `[16] CECI`

**简介**

- CECI 是偏索引和并行处理路线的方法，核心是构建 `Compact Embedding Cluster Index`，将数据图切分成若干 embedding clusters 进行并行处理。
- 其摘要中提到的三个关键技术分别是：
- `BFS-based filtering` 和 `reverse-BFS-based refinement`
- 用 `set intersection` 替代部分边验证
- 基于搜索基数估计提前切分大 cluster

**实验规模**

- 已确认论文发表于 `SIGMOD 2019`。
- 官方摘要明确说明使用了若干真实和合成数据集。
- 实验既评测“列出全部 embeddings”的性能，也评测“枚举前 `1,024` 个 embeddings”的性能。

**实验情况**

- 论文摘要给出的结果很亮眼：平均比已有方法快 `20.4x`，在“前 `1,024` 个 embeddings”任务上平均快 `2.6x`。
- 这说明它在系统性能层面很强，但其优势较多来自索引和并行结构，而不仅仅是搜索过程中的轻量剪枝。

**适合复现程度**

- 优点：如果做出来，实验结果和系统感会很强。
- 风险：索引、cluster、refinement、并行，这些都会显著拉高工程复杂度；对于当前基于 `Python` baseline 起步的项目来说，不是最省风险的选择。

### 4.5 `[17] VF3`

**简介**

- VF3 是 VF 系列中的经典精确匹配算法，重点面向大图和稠密图场景。
- 相比 VF2，VF3 在节点分类、节点访问顺序和 look-ahead 剪枝方面做了增强，因此更适合作为“精确匹配经典路线”的代表方法。

**实验规模**

- 已确认论文发表于 `IEEE TPAMI 2018`。
- 从公开元数据看，论文与 `MIVIA large dense graphs dataset` 相关，实验重点非常明确，就是“大图、稠密图、精确子图同构”。
- 公开实现、资料和讲解相对更多，这一点对课程复现很有帮助。

**实验情况**

- VF3 的优势主要体现在 huge/dense graph 场景下的精确匹配效率提升。
- 它不是最新方案，但胜在经典、稳健、资料多、实现路径清晰。

**适合复现程度**

- 优点：稳妥、成熟、容易找到参考实现和二次资料。
- 风险：其过滤和剪枝策略不一定像 GUP 那样边界清晰，因此做课程要求中的“逐项关闭过滤策略”消融时，需要自行定义拆分粒度。

## 6. 当前推荐顺序
按照“课程要求适配度 + 二人小组实现风险 + 消融可做性”综合考虑，当前推荐顺序如下：

1. `[14] GUP`
2. `[17] VF3`
3. `[13] Efficient Subgraph Matching`
4. `[16] CECI`
5. `[15] BEE`

## 7. 推荐理由

### 7.1 当前首选：`[14] GUP`

- 方法边界清晰，核心创新点集中在 guard
- 很适合做课程要求的过滤策略消融
- 容易与当前仓库中的 baseline matcher 对接
- 容易统计 `partial mappings` 和 `pruned partial mappings`
- 报告和答辩时容易把“为什么有效”讲清楚

### 7.2 当前保底方案：`[17] VF3`

- 资料更丰富，经典方法更稳
- 实现路径较清楚，适合在时间紧时保证项目可落地
- 适合做“稳妥完成”导向的方案

### 7.3 当前高风险高收益方案：`[13]`

- 技术含量高，做出来后论文复现深度会更强
- 但实现跨度较大，不是当前阶段最稳的选择

## 8. 最终结论
- 最终选择：`[14] GUP`
- 选择理由：`guard-based pruning` 的方法边界清晰，最适合拆分成可开关模块，最容易对齐课程要求中的 `partial mappings / pruned partial mappings` 统计，也最适合二人小组在有限周期内完成“实现 + 实验 + 消融 + 汇报”。
- 备选方案：`[17] VF3`
- 放弃 `[13]` 作为首选的原因：技术跨度较大，实现风险高于当前项目节奏
- 放弃 `[15]` / `[16]` 作为首选的原因：更偏复杂结构或并行/索引工程，不是当前最稳妥选择

### 9.3 对 `VF3` 的补充说明
- `VF3` 有官方源码入口，且官方页面明确说明源码和 Linux 可执行文件可获取。
- 许可证为 `MIT`。
- 但获取方式不是公开 GitHub 仓库，而是需要先在作者实验室网站注册，再通过下载页面申请获取源码。
- 因此，`VF3` 更适合作为“若需要现成代码，则优先考虑”的保底方案。
